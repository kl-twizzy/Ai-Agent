## О проекте

`AI Agent` — это браузерный агент, который управляется текстовыми командами на естественном языке.

Вместо ручного взаимодействия с сайтами пользователь может написать:

- `Открой сайт ЦБ РФ и найди курс доллара`
- `Зайди на YouTube и найди видео про квантовую физику`
- `Открой маркетплейс и найди рабочий ноутбук до 70000 рублей`

После этого агент:

1. интерпретирует задачу;
2. определяет следующий шаг;
3. вызывает нужный инструмент;
4. управляет браузером;
5. получает новое состояние страницы;
6. повторяет цикл, пока задача не будет выполнена или пока не встретит внешний блокер.

---

## Какую задачу решает проект

Современные сайты требуют от пользователя большого количества ручных действий:

- открыть нужный сайт;
- найти нужный раздел;
- ввести поисковый запрос;
- выбрать подходящий результат;
- проверить ограничения, цену или характеристики;
- перейти дальше по сценарию.

Мы решаем задачу **универсального управления браузером через AI-агента**, чтобы пользователь мог формулировать цель на человеческом языке, а система сама выполняла навигацию и действия в браузере.

Главная идея проекта:

- не писать отдельный скрипт под каждый сайт;
- не зашивать поведение жестко в код;
- использовать LLM как reasoning-слой;
- использовать MCP-style tools как слой действий;
- строить универсальный agent loop.

## Возможности

- Управление браузером через естественный язык
- Пошаговый agent loop: `LLM -> MCP -> Browser`
- Работа с реальным браузером через Playwright
- Запуск в локальном видимом режиме
- Многошаговые пользовательские сценарии
- Поддержка переходов по страницам, поиска, кликов, ввода текста, скролла и навигации
- MCP-style слой инструментов для вызова браузерных действий
- React frontend для запуска задач и просмотра статуса
- FastAPI backend для orchestration и API
- Docker-конфигурация для контейнерного запуска

## Архитектура

Проект построен как **agentic runtime**, а не как набор заранее прописанных сценариев под несколько сайтов.

### Основной поток выполнения

1. Пользователь отправляет задачу через frontend.
2. Backend создает execution и запускает agent runtime.
3. Агент получает текущее состояние страницы.
4. LLM выбирает следующий `tool_call`.
5. MCP client передает вызов в MCP server.
6. MCP server вызывает browser tool.
7. Браузер выполняет действие.
8. Агент заново читает состояние страницы и продолжает цикл.

### Ключевая связка

- `frontend` — пользовательский интерфейс
- `main.py` — FastAPI API и orchestration
- `browser/agent.py` — основной agent loop
- `browser/llm.py` — planner и tool-calling логика
- `mcp_server.py` — MCP server
- `browser/mcp_tools.py` — каталог browser tools
- `browser/browser.py` — низкоуровневое управление браузером через Playwright
- `models.py` — схемы данных, шаги, отчеты, metadata

### Внутренний runtime

Проект использует модель:

`LLM -> tool_call -> MCP client -> MCP server -> browser tool`

Это означает, что языковая модель не управляет браузером напрямую и не исполняет действия “магически”. Она выбирает следующий инструмент и аргументы, а выполнение идет через отдельный и расширяемый слой инструментов.

```text
├─ requirements.txt
├─ run.py
└─ test.py
```

### Что за что отвечает

- `browser/agent.py`  
  Основной цикл агента: observe, decide, call tool, verify.

- `browser/browser.py`  
  Работа с браузером через Playwright: вкладки, переходы, клик, ввод, скролл, поиск элементов.

- `browser/llm.py`  
  Работа с LLM: prompts, planner, tool-call normalization, вызовы модели.

- `browser/mcp_tools.py`  
  Каталог инструментов, доступных агенту через MCP-layer.

- `mcp_server.py`  
  MCP server, который валидирует tool calls и вызывает browser tools.

- `main.py`  
  Backend API, execution lifecycle, health-check, frontend serving.

- `models.py`  
  Схемы данных: шаги агента, constraints, отчеты, metadata.

- `frontend/`  
  Интерфейс для ввода задачи, просмотра статуса и результата.

## Как запустить проект

### 1. Зависимости

Нужно установить:

- Python 3.11+ или совместимую версию
- Node.js + npm
- Playwright browser binaries

### 2. Установка backend-зависимостей

```bash
pip install -r requirements.txt
```

### 3. Установка браузера для Playwright

```bash
playwright install chromium
```

### 4. Установка frontend-зависимостей

```bash
cd frontend
npm install
```

### 5. Сборка frontend

```bash
npm run build
cd ..
```

### 6. Запуск проекта локально

```bash
python -m uvicorn main:app --host 127.0.0.1 --port 8000
```

После запуска открой:

```text
http://localhost:8000
```

### 7. Пример локального запуска на Windows

```powershell
cd c:\Users\kiril\OneDrive\Документы\Hackthon2\project
pip install -r requirements.txt
playwright install chromium
cd frontend
cmd /c npm install
cmd /c npm run build
cd ..
python -m uvicorn main:app --host 127.0.0.1 --port 8000
```

### 8. Запуск через Docker

```bash
docker compose up --build
```

После этого проект будет доступен по адресу:

```text
http://localhost:8000
```

## Как пользоваться решением

После запуска пользователь открывает интерфейс и вводит задачу в текстовое поле.

Дальше система:

- запускает execution;
- начинает agent loop;
- показывает статус выполнения;
- отображает результат, ссылки и шаги.

### Примеры команд

- `Открой сайт ЦБ РФ и найди официальный курс доллара`
- `Зайди на YouTube и найди видео про квантовую физику`
- `Открой РБК и найди свежие новости про искусственный интеллект`
- `Открой маркетплейс и найди рабочий ноутбук до 70000 рублей`
- `Открой сайт и найди нужный раздел с тарифами`

### Пример работы через API

Запуск задачи:

```bash
curl -X POST http://localhost:8000/execute \
  -H "Content-Type: application/json" \
  -d "{\"task\":\"Открой маркетплейс и найди рабочий ноутбук до 70000 рублей\"}"
```

Проверка статуса:

```bash
curl http://localhost:8000/executions/<execution_id>
```

Проверка health endpoint:

```bash
curl http://localhost:8000/health
```

## Почему проект интересный

Это не просто автокликер и не набор сценариев под 2-3 сайта.

Проект показывает архитектурный подход к созданию **универсального AI-агента для браузера**, где reasoning вынесен в LLM, действия вынесены в MCP-tools, а runtime организован как наблюдение, выбор инструмента, исполнение и повторный анализ состояния страницы.
